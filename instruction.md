Создание API Gateway для единой системы генерации контента
Контекст задачи: Проект состоит из множества микросервисов (SportZavod, content-zavod, Orchestrator, Dashboard API). Требуется спроектировать добавление единого API-шлюза (API Gateway), опираясь на актуальные тренды индустрии 2026 года.

🌍 Анализ: API Gateway в 2026 году
Что такое API Gateway и зачем он нужен?
В 2026 году API Gateway окончательно трансформировался из простого балансировщика нагрузки и роутера в интеллектуальный слой управления (Intelligence Control Layer) цифровой экосистемой.

Основные задачи шлюза сегодня:

Single Point of Entry (Единая точка входа): Скрытие всей внутренней микросервисной сетевой инфраструктуры от публичного доступа. Клиенты (UI, внешние интеграции) общаются только с портом шлюза.
Shift-Left Security & Auth: Перенос проверок безопасности (JWT, OAuth, DPoP, валидация tenant_id) из кода каждого микросервиса на уровень шлюза. Если токен невалиден, запрос даже не дойдёт до SportZavod.
Защита от ИИ-ботов и Rate Limiting: С развитием агентных систем шлюзы научились отличать легитимный ИИ-трафик от вредоносного скрапинга и DDoS за счёт поведенческого анализа.
Управление "AI Sprawl" (разрастанием ИИ-вызовов): Шлюз помогает мониторить объём обращений к дорогим эндпоинтам (например, вызовам LLM и HeyGen внутри SportZavod) и управлять лимитами для каждого tenant_id.
Best Practices для такого сервиса (на основании трендов 2026 года)
API-Native Architecture & Contract-Driven Design: Шлюз должен конфигурироваться декларативно на основе спецификаций (OpenAPI), без ручного программирования каждого роута.
Centralized Observability: Шлюз обязан отслеживать распределённую трассировку (Distributed Tracing). Вы должны видеть в логах полный путь запроса: от Dashboard через Gateway в Orchestrator и далее в Redis/Workers.
Fine-Grained Authorization: Внедрение политик доступа по tenant_id на уровне шлюза. Шлюз проверяет JWT, читает из него ID арендатора и передаёт в микросервисы в виде безопасного внутреннего заголовка (например, X-Tenant-ID).
Отказ от "Изобретения велосипеда": Не рекомендуется писать Gateway с нуля на Python/Node.js, так как готовые Cloud-Native решения (Traefik, Kong, APISIX, Envoy) решают эту задачу за миллисекунды без накладных расходов памяти и легко интегрируются в Docker/Kubernetes.
🛠 Предлагаемый план интеграции для микросервисов
Текущая архитектура из INTEGRATION_PLAN 2.md подразумевает, что каждый компонент торчит наружу или Dashboard напрямую обращается к разным портам (:8000, :8001, :8002, :3001).

С API Gateway все клиенты (Dashboard UI, мобильные устройства) будут стучаться только на один порт (например, :8080), а Gateway маршрутизирует трафик внутри приватной Docker-сети.

Предлагаемые изменения
Обновление будет разбито на следующие блоки:

[MODIFY] docker-compose.yml (Маршрутизация)
Скрытие портов 8000, 8001, 8002, 3001 из внешней сети (убираем привязку ports: "8000:8000").
Добавление сервиса api-gateway (порт 8080:80).
Добавление меток (labels/конфига) маршрутизации:
/api/sportzavod/ -> внутренний контейнер sportzavod:8000
/api/orchestrator/ -> внутренний контейнер orchestrator:8001
/api/contentzavod/ -> внутренний контейнер contentzavod:8002
/api/dashboard/ -> внутренний контейнер dashboard-api:3001
[MODIFY] atome-studio-ui 2/apps/web/
Изменение конфигурации Axios / Fetch во фронтенде. Единый API_BASE_URL = http://localhost:8080/api/.
Переход адаптеров (sportzavod.adapter.ts, farm.adapter.ts, contentzavod.adapter.ts) на использование единого домена шлюза с соответствующими префиксами.
[MODIFY] Сервисы: SportZavod, ContentZavod, Orchestrator
Адаптация JWT middleware. Микросервисам больше не нужно самостоятельно валидировать подпись JWT из базы (если это делает шлюз). Они могут доверять заголовку X-Tenant-ID, который передал шлюз, так как к ним нельзя обратиться в обход шлюза. (Это опционально, можно оставить двойную проверку).
❓ Open Questions
[!QUESTION]

Нужно ли на уровне API Gateway сразу заложить политики Rate Limiting (например, не больше 100 запросов в минуту на один tenant_id), или пока реализовать только маршрутизацию и аутентификацию?
Согласны ли вы использовать Traefik в качестве базового шлюза для конфигурации docker-compose?
Нужна ли поддержка HTTPS прямо сейчас на уровне шлюза (авто-обновление сертификатов Let's Encrypt), если сервис будет разворачиваться на публичном IP, или останемся в рамках HTTP (localhost)?
